# Milestone v0.1.0-rc.1: Flexible Layout Resolution

**Status:** ✅ SHIPPED 2026-04-07
**Phases:** 1-5
**Total Plans:** 10

## Overview

This milestone added flexible storage-layout resolution to `changes` without compromising authoritative writes. The shipped release supports both `xdg` and `home` layouts for global and repo scopes, adds manifest-backed authority detection, exposes `doctor`-based inspection and migration guidance, and verifies the rollout boundary between manifest-backed and legacy repositories.

## Milestone Summary

**Key Accomplishments**

- Locked the governing `xdg`/`home` layout proposal, command contract, and requirement-to-decision traceability before implementation began.
- Built a shared resolver core for global and repo scopes with strict manifest validation, symbolic path preservation, and deterministic repo-init layout selection.
- Enforced authority and safety rules so normal commands fail loudly on true ambiguity, collapse canonical equivalents, and never dual-write.
- Shipped `changes doctor`, JSON and explain output, migration-brief generation, and explicit `init` / `init global` layout UX with state-ignore handling.
- Verified the focused precedence model and rollout boundary: manifest-backed repositories operate normally, while legacy repositories fail cleanly and remain diagnosable.

**Key Decisions**

- `xdg` and `home` are the only supported layout styles.
- XDG remains the default layout style.
- `doctor` is the single inspection and migration-brief command family.
- Only manifest-backed layouts are operationally valid for ordinary commands.
- Layout manifests are structural, symbolic, and low-churn.

**Issues Resolved**

- Path resolution is now centralized in a shared global/repo resolver instead of scattered helper assumptions.
- Repo and global layout selection now have deterministic precedence and explicit operator-facing diagnostics.
- Migration help and authority inspection are available without introducing automatic destructive migration or dual writes.

**Issues Deferred**

- Validate a completed migration result against the expected source and destination layouts.
- Support future directory schema revisions beyond the first flexible-layout rollout.

**Technical Debt Incurred**

- Legacy repositories without `layout.toml` still require `doctor`-guided repair or migration before ordinary commands operate.
