# Requirements Archive: v0.1.0-rc.2

**Archived:** 2026-04-07
**Core Value:** `changes` must make release metadata location predictable, inspectable, and safe even when multiple supported storage layouts are possible.

## Shipped Requirements

### Legacy Layout Repair

- [x] **REPAIR-01**: `changes doctor` can repair a legacy repo-local layout by writing the authoritative layout manifest for the preferred supported candidate
- [x] **REPAIR-02**: `changes doctor --scope repo --repair` refuses to proceed when repo authority is ambiguous instead of guessing
- [x] **REPAIR-03**: successful repair leaves the repo operational for ordinary commands without requiring manual manifest creation
- [x] **REPAIR-04**: repair output explains what was repaired and which authoritative layout is now active

### Safety and Scope

- [x] **SAFE-01**: repair only stamps or repairs one authoritative layout; it does not migrate data or dual-write
- [x] **SAFE-02**: repair preserves the existing repo-local state ignore rule for the authoritative layout
- [x] **SAFE-03**: documentation explains when to use repair versus migration prompts

## Deferred Requirements For Next Milestone

- **MIGR-05**: `changes` can validate a user-completed migration result against the expected source and destination layouts
- **MIGR-06**: `changes` can support future directory schema revisions beyond the first flexible-layout rollout
- **REPAIR-05**: `changes` can offer a comparable repair path for supported global legacy layouts if operator demand justifies it

## Requirement Outcomes

- Validated and shipped: REPAIR-01..04, SAFE-01..03
- Adjusted during milestone: none
- Dropped: none

## Out of Scope At Ship Time

| Feature | Reason |
|---------|--------|
| Automatic data migration between `xdg` and `home` layouts during repair | Repair is intentionally narrow and should not rewrite operator data |
| Global-layout repair automation | Repo-local repair was the immediate operator pain point and the fastest safe closeout |
| Warning-suppression config for non-authoritative sibling layouts | This milestone focused on repair, not on adding suppression policy or persistence |
