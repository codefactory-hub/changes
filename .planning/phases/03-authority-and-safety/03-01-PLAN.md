<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem manifests -> resolver core | Untrusted `layout.toml` files and detected repo directories cross into authority selection logic |
| resolver results -> authority helpers | Candidate evidence and sibling states become operational allow/deny decisions for later commands |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | T | `internal/config/manifest.go` | mitigate | Reject unsupported `schema_version`, mismatched `scope` or `style`, and malformed symbolic paths so invalid manifests never become operational authorities. |
| T-03-02 | T | `internal/config/resolution.go` | mitigate | Group operational candidates by canonical physical target so equivalent layouts do not create false ambiguity and distinct operational layouts do not collapse silently. |
| T-03-03 | R | `internal/config/authority.go` | mitigate | Preserve typed `AuthorityError` status plus competing-candidate data so later command and doctor surfaces can explain exactly why a scope failed. |
| T-03-04 | D | `internal/config/resolution.go` | mitigate | Return structured sibling warnings instead of failing every mixed-candidate repo, allowing safe continuation only when exactly one operational authority exists. |
</threat_model>
