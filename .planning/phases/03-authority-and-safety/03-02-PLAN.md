| authority helpers -> app write flows | Structured authority outcomes determine whether init, create, and release may mutate managed repo state |
| app results -> CLI stderr/stdout | Structured warnings cross from service layers into human-visible CLI output |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | T | `internal/config/config.go` | mitigate | Require repo authority checks before loading authoritative config or allowing managed writes so ambiguous and legacy-only layouts never become silent write targets. |
| T-03-06 | T | `internal/app/app.go` and `internal/app/init.go` | mitigate | Recheck `RequireRepoWriteAuthority` at each write-capable entry point instead of assuming an earlier read result is still safe, per D-14. |
| T-03-06a | T | `internal/app/init.go` | mitigate | Resolve global layout through `CheckScopeAuthority` before consuming `[repo.init]` defaults so ambiguous or invalid global layouts never silently disappear during `init`. |
| T-03-07 | R | `internal/cli/app.go` | mitigate | Print structured warnings on stderr for ordinary commands so operators can see that non-operational sibling layouts still exist beside the authoritative target. |
| T-03-08 | I | `internal/cli/app.go` | accept | Warning output intentionally names the affected scope and sibling path because the operator already has local filesystem access and Phase 4 doctor details depend on this visibility. |
