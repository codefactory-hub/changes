<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flags -> doctor request | Untrusted operator input chooses scope, destination style, optional home path, and optional output path |
| filesystem layouts -> doctor service | Resolver reads untrusted manifests and layout directories to build diagnostics and migration facts |
| migration brief -> output file/stdout | Generated Markdown leaves the process boundary and may be consumed by humans or external LLM tools |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | T | `internal/app/doctor.go` | mitigate | Build inspection on the typed resolver and authority contracts from Phases 2-3 so malformed manifests and ambiguous candidates stay classified, not normalized into false authority. |
| T-04-02 | I | `internal/app/doctor.go` | mitigate | Limit migration prompt content to deterministic layout metadata, inventory facts, and verification steps; do not include uncontrolled file bodies or execute migration actions. |
| T-04-03 | T | `internal/cli/app.go` | mitigate | Validate `--scope`, `--to`, and `--home` combinations explicitly, and only write to disk when the operator passes `--output PATH`. |
| T-04-04 | R | `internal/cli/app.go` | mitigate | Preserve explicit ambiguity/warning/status details in CLI output so operators can audit why a scope resolved, failed, or needs migration help. |
</threat_model>
