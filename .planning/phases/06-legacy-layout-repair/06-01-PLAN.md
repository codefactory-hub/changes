<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flags -> repair request | Untrusted operator input controls repair mode, scope, and command combinations |
| resolver candidate set -> repair selection | Repair logic interprets on-disk legacy layouts and must not guess when multiple candidates are plausible |
| repair write -> repo filesystem | The repair flow mutates repo metadata by writing `layout.toml` and possibly `.gitignore` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | T | `internal/app/doctor.go` | mitigate | Determine repairability from the full repo candidate set, not only `Preferred`, and fail when more than one legacy candidate could be stamped. |
| T-06-02 | T | `internal/config/manifest.go` | mitigate | Reuse the same symbolic repo-manifest encoding rules as init so repair cannot silently stamp a different layout contract. |
| T-06-03 | E | `internal/cli/app.go` | mitigate | Restrict `--repair` to repo scope and reject combinations that would blur repair with inspection or migration workflows. |
| T-06-04 | R | `internal/cli/app.go` | mitigate | Emit explicit success/failure text naming the repaired authoritative layout or the reason repair was refused, so operators can audit what happened. |
| T-06-05 | T | repo filesystem writes | mitigate | Limit repair mutation to manifest stamping and authoritative `.gitignore` preservation; rerun authority checks after the write before reporting success. |
</threat_model>
